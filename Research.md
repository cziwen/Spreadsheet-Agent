# Market & Product Research

## Spreadsheet AI工具当前的限制
- 缺少对表格数据的语义理解，例如列与列或表与表之间的依赖关系、业务逻辑、验证规则，这些不仅体现在数字、行、标题里，也体现在函数里
- 现有工具似乎套用代码逻辑“编写 → 测试 → 调试 → 迭代”，但表格数据是需要探索的，并不是线性的。可以考虑多条探索道路同时进行，采用“执行 → 验证”流程，并保持上下文管理
- 缺少多表格思维，不能只读当前表格，需要考虑某工作/项目下的所有表格（类似于code agent读整个codespace）
- 缺少重构项目的功能，缺少推理能力，等等

## Define "Cursor for Spreadsheet"

### 产品形态

**Persona 1: 运营**
- 有业余数据分析能力，熟悉Excel/Sheets
- 工作内容：追踪客户、订单、转化、物流，汇总数据、找问题、做报告
- 痛点
    - 数据一致性难维护，多表格不同步，例如订单+物流+客户
    - 没有快捷方式可以探索更复杂的请求，例如跨表格分析和汇总

**Persona 2: 数据分析师**
- 专业的数据分析能力（SQL/Python）
- 工作内容：数据清理、跨表关联、假设检验、制作dashboard
- 处理多个数据源、多张表格的复杂分析
- 懂SQL/Python，但不想为简单探索写代码
- 痛点
    - 大量重复且耗时的工作容易出现遗漏，例如检验数据质量
    - 做多张表格的复杂分析时要维护关联逻辑和变换规则，容易出ug
    - 无法追溯/验证数据错误，例如当一个数字有问题时，很难追踪它来自哪里、经过了哪些处理

### 核心功能

#### 1. 智能数据关系和跨表分析

**用户在做什么：**
运营："按渠道看，近3个月哪个渠道的新客留存最好？"

**以前怎么做、哪里痛：**
- 手动在订单表中加列判断"是否留存"（重复计算多次）
- 手动复制客户表的渠道信息到订单表，容易出错
- 如果关联逻辑改变（比如改成4个月），要重新所有操作

**用产品怎么做：**
```
用户操作流程：
1. 选中订单表 → 右键或AI菜单 "分析"
2. 自然语言提问："按客户渠道统计，3个月内重复购买的客户占比"
3. AI系统：
   - 自动识别订单表和客户表的关系（通过customer_id）
   - 理解"留存"的含义（自动检查是否有重复购买）
   - 生成一个新的统计表，展示按渠道的留存率 + 趋势

4. 用户审视结果 → 修改参数（比如改成4个月）→ AI自动重新计算
```

**技术支撑：**
- 语义表理解：自动识别表中每一列的含义（customer_id、channel、created_at等）、数据类型、分布
- 跨表关系图：建立工作簿内所有表的关系图（哪些表可以通过什么字段关联）
- 多步Agent：将自然语言分解成多步操作（join → filter → aggregate → format）
- 数据质量检查：在执行前自动检查关联字段的有效性（是否有空值、重复、格式不一致）

---

#### 2. 数据质量诊断和自动修复

**用户在做什么：**
数据分析师：准备重要的业务报告，需要确保数据干净可信

**以前怎么做、哪里痛：**
- 手动scroll表格，eye-ball式地找问题
- 写SQL查询来找重复、缺失、异常值
- 发现问题后，手动修复或问数据工程师
- 不知道有多少隐藏的小问题

**用产品怎么做：**
```
用户操作流程：
1. 选中数据范围 → AI菜单 "检查数据质量"
2. AI自动扫描并生成诊断报告：
   - 缺失值：哪些列有空值（百分比）、是否可接受
   - 格式不一致：日期列有3种格式、电话号码有/没有前缀
   - 离群值：销售额某行异常高、但看起来是真数据（基于域知识）
   - 重复记录：客户表有2个customer_id为同一个人
   - 关键列缺乏约束：应该是正数的列有负数

3. 用户逐条审视 → AI建议修复方案
   - "这一列的空值是数据收集问题，建议填0还是删除？"
   - "这个日期格式看起来是数据导入错误，建议转换"
   - "这3行销售额看起来不真实，建议标记为outlier还是删除？"

4. 用户确认 → AI自动应用修复，或生成清理脚本供用户复查
```

**技术支撑：**
- 数据分布分析：统计每列的类型、范围、分布（正态分布、长尾等）
- 异常检测：基于统计和机器学习识别离群值（不仅是3σ，还考虑业务上下文）
- 格式识别：自动识别日期、电话、邮箱等常见格式的变体
- 启发式规则：基于列名、历史数据推断数据验证规则（如"金额列应该 > 0"）

#### 3. 路径记忆和场景建模

**用户在做什么：**
运营：做"假设分析"，如果我们提高渠道A的投入预算，会怎样？

**以前怎么做、哪里痛：**
- 手动复制表格建立"当前"和"场景1"两个版本
- 修改参数，手动重新计算所有依赖的指标
- 对比两个版本，用notes记录假设
- 如果假设改变，很难复现之前的逻辑

**用产品怎么做：**
```
用户操作流程：
1. 用户进行了一系列分析：过滤 → 分组 → 计算 KPI
2. AI自动记住整个分析路径的上下文
3. 用户说："保存这个场景为'乐观估计'"
4. 用户改参数（比如成本增加20%），AI自动：
   - 理解哪些计算依赖于这个参数
   - 自动重新计算下游指标
   - 对比"当前"和"乐观估计"场景

5. 用户可以创建多个场景（乐观、保守、基线）
6. AI生成对比分析：
   - 哪个场景下哪个指标受影响最大
   - 在什么条件下某个假设才值得投入
```


**技术支撑：**
- 公式依赖图：建立表格中所有公式的依赖关系图（A列依赖于B和C）
- 参数识别：自动识别哪些值是参数（应该可以改），哪些是结果（应该跟随计算）
- 增量计算：当参数改变时，只重新计算依赖的单元格，而不是全表重算
- 上下文保留：维持用户的分析历史，理解"我在做假设分析"这个意图

### 非目标（Non-Goals）

1. **多用户协作**
   - 这是Sheets和Excel的基础功能，而且多用户冲突管理会复杂化数据理解逻辑。我们专注"个人AI助手"而不是"团队"

2. **BI仪表板和数据可视化**
   - Tableau、Google Data Studio已经做得很好。我们专注"数据探索和准备"

4. **预测和机器学习模型**
   - 我们的目标是"帮助用户理解已有数据"，而不是"预测"。这避免了模型可解释性和商业风险的复杂性

5. **写代码的IDE功能**
   - 我们的目标是让非技术用户也能做复杂分析，不是写Python笔记本

6. **与外部数据源的实时同步**
   - 用户应该明确管理数据更新时机，这避免了复杂的数据一致性问题

### 核心

#### 1. 语义数据索引
- 理解列的含义，不仅仅是名称
- 构建跨表/跨工作簿的数据关系知识图
- 推断数据类型、约束、依赖关系
- 追踪数据源头（这个列来自哪里？）

#### 2. 探索工作流理解
- 维持分析路径的上下文："用户过滤 → 分组 → 计算 → 可视化"
- 主动建议："你在按月份分析 —— 想看年同比对比吗？"
- 从更正中学习："用户撤销了我的建议 —— 为什么？"

#### 3. 多表/多工作簿推理
- 理解数据如何在表之间流动
- 检测不一致："表1有100个客户，表2有95个 —— 缺了什么？"
- 跨工作簿自动关联，无需手动映射

#### 4. 数据质量智能
- 具有域感知能力的自动异常值检测
- 不一致性标记："这个日期列有3种不同的格式"
- 验证规则推断："列X总是0-100之间，但第45行是150"

#### 5. 对话上下文记忆
- "添加利润率列" → 记住利润 = 收入 - 成本
- "现在按地区分组" → 从前面的上下文理解'地区'
- "如果排除异常值呢？" → 知道用户在分析哪一列

#### 6. 内联，而非侧边栏
- AI存在于单元格中，不是独立的聊天窗口
- 基于光标位置的上下文感知建议
- 幻影文本预览（如Cursor的内联建议）
