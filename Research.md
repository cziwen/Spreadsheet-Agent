# Market & Product Research

## Spreadsheet AI工具当前的限制
- 缺少对表格数据的语义理解，例如列与列或表与表之间的依赖关系、业务逻辑、验证规则，这些不仅体现在数字、行、标题里，也体现在函数里
- 现有工具似乎套用代码逻辑“编写 → 测试 → 调试 → 迭代”，但表格数据是需要探索的，并不是线性的。可以考虑多条探索道路同时进行，采用“执行 → 验证”流程，并保持上下文管理
- 缺少多表格思维，不能只读当前表格，需要考虑某工作/项目下的所有表格（类似于code agent读整个codespace）
- 缺少重构项目的功能，缺少推理能力，等等

## Define "Cursor for Spreadsheet"

### 产品形态

**Persona 1: 运营**
- 有业余数据分析能力，熟悉Excel/Sheets
- 工作内容：追踪客户、订单、转化、物流，汇总数据、找问题、做报告
- 痛点
    - 数据一致性难维护，多表格不同步，例如订单+物流+客户
    - 没有快捷方式可以探索更复杂的请求，例如跨表格分析和汇总

**Persona 2: 数据分析师**
- 专业的数据分析能力（SQL/Python）
- 工作内容：数据清理、跨表关联、假设检验、制作dashboard
- 处理多个数据源、多张表格的复杂分析
- 懂SQL/Python，但不想为简单探索写代码
- 痛点
    - 大量重复且耗时的工作容易出现遗漏，例如检验数据质量
    - 做多张表格的复杂分析时要维护关联逻辑和变换规则，容易出bug
    - 无法追溯/验证数据错误，例如当一个数字有问题时，很难追踪它来自哪里、经过了哪些处理

### 核心功能

#### 1. 智能数据关系和跨表分析

**用户在做什么：**
运营："按渠道看，哪个渠道的订单总额最高？"

**以前怎么做、哪里痛：**
- 手动在订单表中加列记录渠道信息（需要从客户表复制）
- 手动用SUMIF或透视表计算每个渠道的总额
- 如果客户表更新，需要重新复制数据
- 如果要看不同时间段，要重复所有操作

**用产品怎么做：**
```
用户操作流程：
1. 自然语言提问："哪个channel的订单总额最高？"
2. AI系统：
   - 自动识别需要订单表（amount字段）和客户表（channel字段）
   - 理解需要通过customer_id关联两个表
   - 自动生成执行计划：join → group by channel → sum amount → sort descending
   - 返回结果：organic渠道订单总额最高
```

**技术支撑：**
- 语义表理解：自动识别每列的含义（customer_id、channel、amount、order_date等）、数据类型、分布
- 跨表关系图：自动发现表间关系（通过customer_id可以关联）
- 多步Agent：将自然语言分解成操作序列（select → join → filter → aggregate → sort → limit）

---

#### 2. 数据质量诊断和自动修复

**用户在做什么：**
数据分析师：准备重要的业务报告，需要确保数据干净可信

**以前怎么做、哪里痛：**
- 手动scroll表格，eye-ball式地找问题
- 写SQL查询来找重复、缺失、异常值
- 发现问题后，手动修复或问数据工程师
- 不知道有多少隐藏的小问题

**用产品怎么做：**
```
用户操作流程：
1. 自然语言 "检查订单表的数据质量"
2. AI自动扫描并生成诊断报告：
   - 缺失值：哪些列有空值（百分比）、是否可接受
   - 格式不一致：日期列有3种格式、电话号码有/没有前缀
   - 离群值：销售额某行异常高、但看起来是真数据（基于域知识）
   - 重复记录：客户表有2个customer_id为同一个人
   - 关键列缺乏约束：应该是正数的列有负数
```

**技术支撑：**
- 数据分布分析：统计每列的类型、范围、分布（正态分布、长尾等）
- 异常检测：基于统计和机器学习识别离群值（不仅是3σ，还考虑业务上下文）
- 格式识别：自动识别日期、电话、邮箱等常见格式的变体
- 启发式规则：基于列名、历史数据推断数据验证规则（如"金额列应该 > 0"）

#### 3. 场景建模

**用户在做什么：**
运营：做"假设分析"，如果我们提高所有订单金额售价10%，会怎样？

**以前怎么做、哪里痛：**
- 手动复制表格建立"当前"和"场景1"两个版本
- 修改参数，手动重新计算所有依赖的指标
- 对比两个版本，用notes记录假设
- 如果假设改变，很难复现之前的逻辑

**用产品怎么做：**
```
用户操作流程：
1. 自然语言 "创建一个场景，订单金额增加10%"
2. AI生成对比分析：
   - 复制新的副本并应用假设
   - 预览对比分析
```


**技术支撑：**
- 公式依赖图：建立表格中所有公式的依赖关系图（A列依赖于B和C）
- 参数识别：自动识别哪些值是参数（应该可以改），哪些是结果（应该跟随计算）
- 增量计算：当参数改变时，只重新计算依赖的单元格，而不是全表重算

### 未来功能 / Future Roadmap

#### 高级分析能力 (Advanced Analytics)

当前系统专注于基础的跨表查询和聚合分析。以下是计划中的高级功能：

**1. 留存分析 (Retention Analysis)**

**用户需求：**
"按渠道看，近7天哪个渠道的新客留存最好？"

**技术挑战：**
- 需要日期算术：计算两个日期之间的天数差
- 需要窗口函数：识别每个客户的"第一次购买"和"第二次购买"
- 需要条件聚合：统计"在7天内重复购买"的客户比例
- 需要自连接：将订单表自己和自己关联，比较同一客户的多次购买

**实现路径：**
1. 添加日期计算操作：date_diff, date_add, extract (month/year/day)
2. 添加窗口函数：ROW_NUMBER, RANK, FIRST_VALUE, LAG/LEAD
3. 添加条件聚合：COUNT_IF, SUM_IF with complex conditions
4. 支持自连接和多次关联同一张表

**2. 同期群分析 (Cohort Analysis)**

根据用户首次购买时间分组，追踪每个同期群随时间的行为变化。

**技术需求：**
- 时间序列变换（按月/周分组）
- 累积指标计算
- 多维度透视表生成

**3. 客户生命周期价值 (Customer Lifetime Value)**

预测客户未来价值，识别高价值客户群体。

**技术需求：**
- 时间窗口聚合（最近30天/90天/1年）
- 趋势计算（增长率、变化率）
- 条件分组和标签生成

**4. 业务指标模板**

为常见业务场景提供预定义的分析模板：
- "新客获取成本" (CAC)
- "月活跃用户" (MAU)
- "客单价" (Average Order Value)
- "复购率" (Repeat Purchase Rate)

用户可以直接问："计算这个月的复购率"，系统理解这是一个模板化指标并自动计算。

### 非目标（Non-Goals）

1. **多用户协作**
   - 这是Sheets和Excel的基础功能，而且多用户冲突管理会复杂化数据理解逻辑。我们专注"个人AI助手"而不是"团队"

2. **BI仪表板和数据可视化**
   - Tableau、Google Data Studio已经做得很好。我们专注"数据探索和准备"

4. **预测和机器学习模型**
   - 我们的目标是"帮助用户理解已有数据"，而不是"预测"。这避免了模型可解释性和商业风险的复杂性

5. **写代码的IDE功能**
   - 我们的目标是让非技术用户也能做复杂分析，不是写Python笔记本

6. **与外部数据源的实时同步**
   - 用户应该明确管理数据更新时机，这避免了复杂的数据一致性问题
